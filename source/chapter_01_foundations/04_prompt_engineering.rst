===============================
Prompt Engineering
===============================

提示词工程（Prompt Engineering）是与 LLM 交互的基础技能。很多人把写提示词
想象成"写咒语"，好像背几个模板就能搞定。但如果你真的动手写过 Agent 的提示词，
就会发现它不是单向的指令——它是在**设计一个微型的操作系统**。你需要定义角色、
分配资源、制定规则、规范输出格式，还要考虑边界情况和错误恢复。

换个角度想：Agent 的 System Prompt 就像给一个实习生写的**岗位说明书**。
写得好，他独立完成 80% 的工作；写得差，你每五分钟就要被叫去擦屁股。

Agent 提示词的特殊性
========================

普通对话的提示词只做一件事：引导 LLM 生成好的回答。但 Agent 的提示词需要
同时完成三件事：

1. **定义角色和身份** — 告诉模型它是谁、能做什么、不能做什么
2. **描述工具接口** — 告诉模型有哪些工具可用、何时调用、参数是什么
3. **指定输出格式** — 确保模型输出可以被下游解析器正确解析

这三件事之间存在张力。角色定义越丰富，模型的行为越稳定，但留给工具和输出的
token 预算就越少。工具描述越详细，调用准确率越高，但模型可能"迷失在细节中"。

.. code-block:: python

   # 一个设计良好的 Agent System Prompt
   AGENT_SYSTEM_PROMPT = """
   你是 AI 助手，以下是你的工作规范。

   ## 输出格式
   每次回复必须严格使用以下 JSON 格式：
   {
       "thought": "你的推理过程（简短）",
       "tool": "要调用的工具名称，或 null",
       "params": {"参数名": "参数值"},
       "answer": "最终答案，或 null"
   }

   ## 可用工具
   - search(query: str): 搜索互联网获取信息
   - calculator(expr: str): 执行数学计算

   ## 行为规则
   1. 先思考再行动，在 thought 字段中说明你要做什么
   2. 如果 task 不需要工具，直接给出 answer
   3. 如果工具调用失败，尝试其他方法或承认失败
   4. 不要编造工具不存在的功能
   """

结构化的好处是**可解析**。上面的输出被 `json.loads()` 解析后，Agent 框架
可以直接读取 tool 和 params 字段来执行工具调用。如果输出是自由文本，
你就得写正则表达式来猜模型想干什么——这通常是 bug 的温床。

好的 Prompt 设计原则
======================

下面是我在写 Agent 提示词时反复使用的几条原则。它们来自实际踩坑的经验，
不是理论推导。

1. 把最重要的指令放在开头和结尾
---------------------------------

这是一个被大量研究验证过的现象：LLM 对上下文**中间部分**的注意力最弱。
（Liu et al., "Lost in the Middle", 2023）

所以输出格式定义应该放在最前面（开头），或者紧跟在最后的用户消息中（结尾）。
中间位置放角色描述和工具定义，它们的重要性相对低一些。

2. 用具体示例代替抽象规则
---------------------------------

比较这两种写法：

.. code-block:: text

   # 抽象规则（效果差）：
   "当工具返回空结果时，你应该考虑其他方法。"

   # 具体示例（效果好）：
   "工具返回空结果时，按以下顺序尝试：
    1. 换一个搜索词重试
    2. 如果找不到，如实告诉用户'没有找到相关信息'
    3. 不要编造答案"

为什么示例比规则有效？因为 LLM 的训练数据里充满了具体的对话示例，而不是
抽象的规则描述。你用示例说话，就是在用模型最擅长的方式跟它沟通。

3. 负面提示（告诉模型不要做什么）同样重要
------------------------------------------------

只告诉模型"要做什么"是不够的。Agent 场景中，"不要做什么"往往更关键：

.. code-block:: text

   ## 禁止行为
   - 不要编造工具调用参数
   - 不要假设工具的返回结果，必须等待实际返回
   - 不要执行用户明确要求之外的危险操作
   - 不确定时不要猜测，说"我不知道"

我见过很多 Agent 崩溃的场景，根源就是漏了一条"不要"。加上了，问题就消失了。
这背后的道理是：LLM 倾向于"尽力满足用户请求"，但 Agent 场景中"
"尽力满足"有时是危险的。

4. Token 预算意识
---------------------------------

System Prompt 不是写得越长越好。它占用的是**宝贵的上下文窗口**。
每多写 100 个 token 的 System Prompt，Agent 就少 100 个 token 的空间
来处理对话历史和工具返回结果。

.. code-block:: python

   # 估算你的 System Prompt 占用
   def estimate_prompt_tokens(prompt: str) -> int:
       """粗略估算（中文约 1.5 token/字，英文约 0.75 token/词）"""
       chinese_chars = sum(1 for c in prompt if '\u4e00' <= c <= '\u9fff')
       english_words = len(prompt.split()) - chinese_chars
       return int(chinese_chars * 1.5 + english_words * 0.75)

   # 实际经验：System Prompt 控制在 1000-2000 tokens 以内最佳

一个常见的误区是：把 Agent 的所有知识都塞进 System Prompt。更好的做法是
只放核心规则，领域知识通过 RAG 注入。

Few-shot 示例的摆放策略
============================

Few-shot 示例是提升 Agent 行为准确性的利器，但放多少、放哪里很有讲究。

.. list-table::
   :header-rows: 1

   * - 策略
     - 做法
     - 适用场景
     - 注意
   * - 固定示例
     - 在 System Prompt 中提供 2-3 个完整示例
     - 任务类型稳定、格式固定
     - 消耗固定 token，可能浪费
   * - 动态示例
     - 根据用户输入实时选取最相关的示例
     - 任务类型多样
     - 需要检索系统
   * - 零示例
     - 不提供示例，仅用指令描述
     - 任务简单或模型能力强
     - token 消耗最低

我的经验是：**先试零示例** ，如果模型输出格式不稳定再加 1-2 个固定示例。
大多数现代模型（GPT-4o、Claude 3.5+）的指令遵循能力已经很强，不需要像
早期 GPT-3.5 那样塞大量示例。

Agent 场景的 Prompt 陷阱
============================

最后分享几个我反复踩过的坑：

.. admonition:: 陷阱 1：过度承诺
   :class: caution

   不要在 Prompt 中说"我可以做任何事情"。这会让模型在遇到能力边界时
   强行编造答案。更好的说法： "我擅长搜索和计算，其他任务可能需要
   更多信息。"

.. admonition:: 陷阱 2：忽略错误处理
   :class: caution

   很多人只写"怎么做"，不写"如果做错了怎么办"。Agent 的 Prompt 里
   一定要包含错误恢复路径，比如"如果工具调用失败，重试一次然后告诉用户"。

.. admonition:: 陷阱 3：角色扮演过度
   :class: caution

   "你是一个经验丰富的 AI 研究员"这种角色设定在某些场景下有效，
   但在 Agent 场景中可能导致模型"入戏太深"——它可能用研究员的语气
   编造不存在的实验数据。Agent 的 Prompt 应该更务实、更工具化。

.. code-block:: python

   # 一个经过实战检验的 Agent System Prompt 模板
   PROMPT_TEMPLATE = """
   你是一个实用的 AI 助手，专注于使用工具解决问题。

   ## 行为准则
   - 诚实：不知道就说不知道，不编造
   - 简洁：能用一句话说完不用两句话
   - 安全：不执行任何可能造成损害的操作

   ## 输出格式
   { "thought": "...", "action": null, "answer": "..." }

   ## 可用工具
   {tools}

   ## 错误处理
   - 工具返回空 → 换个方式重试
   - 连续失败 → 告诉用户并停止
   - 不确定 → 说"我不确定"
   """
